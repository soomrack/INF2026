# Лекция 05. Объектно-ориентированное программирование

## Кратко о лекции

Лекция вводит объектно-ориентированное программирование: объект, класс, инкапсуляцию, наследование, полиморфизм, конструкторы, деструкторы и принцип RAII

## 1. Что такое ООП

ООП — подход к программированию, где программа строится вокруг объектов

Объект объединяет:

- данные
- операции над этими данными

Исторически ООП развивалось с конца 1950-х годов и стало заметным направлением во второй половине 1960-х годов

## 2. Объект

Объект можно понимать как сущность, которая хранит данные и умеет выполнять действия

Пример из реальности:

- термометр хранит температуру
- банковский счёт хранит баланс
- датчик хранит состояние измерения

В программе объект помогает связать данные и функции, которые с этими данными работают

## 3. Класс

Класс — шаблон для создания объектов

Он описывает:

- какие данные есть у объекта
- какие методы доступны
- как объект создаётся
- как объект ведёт себя

Пример:

```cpp
class Thermometer {
public:
    float temperature;

public:
    void get_temperature();
};
```

`Thermometer` — класс, а `top` и `bottom` могут быть разными объектами этого класса

## 4. Методы класса

Метод — функция, принадлежащая классу

```cpp
void Thermometer::get_temperature()
{
    temperature = dht.readTemperature();
}
```

Метод работает с полями конкретного объекта

Поэтому запись:

```cpp
top.get_temperature();
```

означает, что метод вызван именно для объекта `top`

## 5. Инкапсуляция

Инкапсуляция объединяет данные и методы внутри класса и скрывает детали реализации от внешнего кода

Зачем нужна инкапсуляция:

- уменьшает сложность
- защищает данные от неправильного изменения
- позволяет менять реализацию без изменения внешнего кода
- делает объект понятной самостоятельной частью программы

Простая идея: внешний код должен пользоваться объектом через методы, а не лезть напрямую в его внутреннее устройство

## 6. Контроль доступа

В C++ есть уровни доступа:

- `public` — доступно снаружи
- `private` — доступно только внутри класса
- `protected` — доступно внутри класса и его наследников

Пример:

```cpp
class Bank {
private:
    int account;

public:
    void income_salary(int total);
    int get_account() { return account; }
};
```

Поле `account` нельзя менять напрямую, поэтому все операции проходят через методы

## 7. Инкапсуляция в C

В C тоже можно имитировать инкапсуляцию через разделение `.h` и `.c` файлов

В заголовке можно показать только объявление структуры и функции:

```c
struct Thermometer;
void get_temperature(struct Thermometer *thermometer);
```

А внутреннее устройство структуры спрятать в `.c` файле

## 8. Дублирование кода

Без наследования похожие классы приходится описывать заново

Например, обычный термометр и тестируемый термометр могут иметь одинаковую температуру, но второй ещё хранит признак неисправности

Проблема: одинаковые поля и методы начинают повторяться

## 9. Наследование

Наследование позволяет создать новый класс на основе уже существующего

```cpp
class ThermometerTested : public Thermometer {
public:
    bool is_broken;

public:
    void run_self_test();
};
```

`ThermometerTested` получает возможности `Thermometer` и добавляет свои

Зачем нужно наследование:

- уменьшить дублирование
- расширить функциональность
- использовать общий интерфейс для похожих объектов

## 10. Наследование и параметры функций

Если функция принимает объект базового класса, то объект наследника может использоваться как базовый объект

```cpp
void control_temperature(Thermometer thermometer)
{
    if (thermometer.temperature > 50) {
        fan_start();
    }
}
```

Это удобно, когда разные объекты имеют общую основу

## 11. Наследование в C

В C похожее поведение можно имитировать через структуры и приведение типов

Проблема такого подхода:

- слабый контроль типов
- выше риск ошибки
- меньше защиты со стороны компилятора

В C++ наследование встроено в язык и поэтому безопаснее и удобнее

## 12. Принцип Open/Closed

Open/Closed Principle: программные сущности должны быть открыты для расширения, но закрыты для изменения

Практический смысл:

- не нужно постоянно менять базовый класс
- новую функциональность лучше добавлять через наследников
- старый код меньше ломается при расширении программы

## 13. Конструкторы

Конструктор — специальный метод, который вызывается при создании объекта

```cpp
class Bank {
public:
    int account;

public:
    Bank() { account = 0; }
    Bank(int account) : account(account) {}
};
```

Конструктор нужен, чтобы объект сразу получил корректное начальное состояние

## 14. Деструкторы

Деструктор вызывается при уничтожении объекта

```cpp
class Bank {
public:
    int account;
    ~Bank() { account = 0; }
};
```

Деструктор используют для освобождения ресурсов и завершающих действий

## 15. RAII

RAII — принцип, при котором ресурс привязан ко времени жизни объекта

Идея:

- ресурс захватывается в конструкторе
- ресурс освобождается в деструкторе

Пример:

```cpp
class Heap {
private:
    char* mem;

public:
    Heap(size_t size) { mem = malloc(size); }
    ~Heap() { free(mem); }
};
```

Когда объект выходит из области видимости, память освобождается автоматически

## 16. Полиморфизм

Полиморфизм — способность разных объектов по-разному реагировать на один и тот же вызов

В C++ есть два важных варианта:

- перегрузка функций
- виртуальные методы

## 17. Полиморфизм в параметрах

Можно создать несколько функций с одним именем, но разными параметрами

```cpp
void print_temperature(Thermometer thermometer);
void print_temperature(ThermometerTested thermometer);
```

Какая функция будет вызвана, определяется на этапе компиляции

## 18. Полиморфизм в методах

Для позднего связывания используют виртуальные методы

```cpp
class Thermometer {
public:
    virtual void print_temperature();
};

class ThermometerTested : public Thermometer {
public:
    void print_temperature() override;
};
```

Если метод виртуальный, то конкретная версия метода выбирается во время выполнения программы

## 19. Позднее связывание

Разница:

- полиморфизм в параметрах определяется при компиляции
- полиморфизм в виртуальных методах определяется при выполнении

Позднее связывание нужно, когда программа заранее не знает точный тип объекта, но должна вызвать правильный метод

## 20. Главные идеи ООП

Ключевые аспекты ООП:

- инкапсуляция
- наследование
- полиморфизм

Что решает ООП:

- скрывает сложность
- уменьшает дублирование
- помогает расширять программу
- связывает данные и поведение
- упрощает работу с большими проектами

## Итог лекции

ООП помогает строить программу из самостоятельных объектов. Классы описывают структуру объектов, инкапсуляция скрывает детали, наследование уменьшает дублирование, а полиморфизм позволяет работать с разными объектами через общий интерфейс
