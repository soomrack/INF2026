# Блок 6. Object-oriented programming

## Вопрос 40. Что такое объектно‑ориентированное программирование?

Объектно‑ориентированное программирование, или ООП, — это парадигма, при которой программа строится вокруг объектов, объединяющих данные и операции над ними. ООП помогает описывать программу через сущности предметной области, а не только через набор функций. Состояние объекта хранится в полях, а поведение задаётся методами.

Главные идеи ООП — инкапсуляция, наследование и полиморфизм. Инкапсуляция скрывает внутреннее устройство объекта, наследование позволяет создавать новые классы на основе существующих, а полиморфизм даёт возможность работать с разными объектами через общий интерфейс. ООП полезно в больших программах, потому что помогает уменьшать дублирование, скрывать сложность, задавать границы ответственности и расширять код без постоянного изменения уже работающих частей.

Простой пример из предметной области: в программе теплицы объект `Thermometer` может хранить текущую температуру и уметь возвращать её через метод. В коде мы работаем не с отдельными разрозненными переменными, а с сущностью, у которой есть состояние и поведение. Типичная ошибка — думать, что ООП означает только «использовать классы»; важнее правильно выделять объекты и их ответственность. Коротко: ООП организует программу вокруг объектов, которые объединяют данные и действия над ними.

## Вопрос 41. Что такое класс и объект?

Класс — это пользовательский тип данных, шаблон или описание для создания объектов. Он задаёт, какие поля есть у объекта, какие методы доступны, как объект создаётся, уничтожается и как он должен себя вести.

Объект — это конкретный экземпляр класса, существующий во время выполнения программы. Например, класс `Thermometer` может описывать общие свойства термометра, а объекты `top` и `bottom` будут двумя конкретными термометрами со своим состоянием. Класс отвечает на вопрос «какой тип сущности описан», а объект — «какой конкретный экземпляр этого типа сейчас существует в памяти».

У объектов одного класса одинаковый набор полей и методов, но значения полей могут различаться. Поэтому класс можно сравнить с чертежом, а объект — с конкретным изделием по этому чертежу. В C++ класс сам по себе не хранит значения обычных полей для всех объектов сразу; память под поля появляется у каждого созданного объекта.

```cpp
class Thermometer {
public:
    double temperature = 0.0;
};

Thermometer room;
room.temperature = 23.5;
```

Здесь `Thermometer` — класс, а `room` — объект этого класса. У другого объекта этого же класса может быть своё значение температуры. Коротко: класс — описание типа, объект — конкретный созданный экземпляр.

## Вопрос 42. Наследование

Наследование — это механизм ООП, при котором новый класс создаётся на основе уже существующего. Исходный класс называют базовым, а новый — производным. Наследование нужно, когда несколько классов имеют общую основу, но отличаются деталями. Производный класс получает доступные поля и методы базового класса и может добавлять собственные данные и поведение.

Например, класс `ThermometerTested` может наследоваться от `Thermometer` и дополнительно хранить признак неисправности `is_broken` и метод самопроверки. Наследование уменьшает дублирование кода и позволяет использовать общий интерфейс для похожих объектов. В C++ наследование обычно записывается как `class Child : public Parent`, если нужно выразить отношение «является разновидностью».

```cpp
class Sensor {
public:
    double value = 0.0;
};

class TemperatureSensor : public Sensor {
public:
    bool overheated = false;
};
```

`TemperatureSensor` является разновидностью `Sensor`, поэтому получает поле `value` и добавляет своё поле `overheated`. Типичная ошибка — применять наследование там, где нет отношения «является разновидностью»; в таких случаях часто лучше использовать композицию, то есть включить один объект внутрь другого как поле. Например, автомобиль не является двигателем, он содержит двигатель. Коротко: наследование позволяет описать новый класс на основе общего базового класса, когда между классами есть отношение «is-a».

## Вопрос 43. Полиморфизм (два вида)

Полиморфизм — это способность одного интерфейса работать с разными типами данных или объектами. Он решает задачу единообразной работы с разными сущностями без постоянных проверок конкретного типа. В C++ важны два вида полиморфизма: статический и динамический.

Статический полиморфизм определяется на этапе компиляции. К нему относятся перегрузка функций и операторов, а также шаблоны, когда один обобщённый код работает с разными типами. Динамический полиморфизм определяется во время выполнения и реализуется через виртуальные методы. Если базовый класс содержит `virtual`‑метод, а производный класс переопределяет его с `override`, то при вызове через указатель или ссылку на базовый класс будет выбрана версия метода фактического объекта. Это называется поздним связыванием.

```cpp
class Device {
public:
    virtual void print_status() const {
        std::cout << "device";
    }
};

class Pump : public Device {
public:
    void print_status() const override {
        std::cout << "pump";
    }
};
```

Если вызвать `print_status` через ссылку на `Device`, но фактически объект будет `Pump`, выполнится версия из `Pump`. Типичная ошибка — забыть `virtual` в базовом классе, тогда динамического полиморфизма не будет. Ещё один важный нюанс C++: если объекты производных классов удаляются через указатель на базовый класс, деструктор базового класса обычно должен быть виртуальным. Коротко: полиморфизм позволяет обращаться к разным объектам через общий интерфейс.

## Вопрос 44. Инкапсуляция

Инкапсуляция — это объединение данных и методов, работающих с этими данными, внутри одного класса, а также скрытие деталей реализации от внешнего кода. Внешний код должен пользоваться объектом через понятный интерфейс, а не напрямую изменять его внутреннее состояние.

Например, поле банковского счёта лучше сделать `private`, а операции пополнения и получения баланса оформить как `public`‑методы. Тогда объект сам контролирует корректность изменений. Инкапсуляция уменьшает сложность, защищает данные от неправильного использования и позволяет изменить внутреннюю реализацию класса без изменения кода, который этим классом пользуется.

```cpp
class Account {
private:
    int balance = 0;

public:
    void deposit(int amount) {
        if (amount > 0) {
            balance += amount;
        }
    }

    int get_balance() const {
        return balance;
    }
};
```

Внешний код не может напрямую записать отрицательный баланс, а должен пользоваться методами класса. Здесь важен инвариант — условие корректности объекта, например баланс не должен изменяться произвольно. Типичная ошибка — делать все поля `public` и тем самым терять контроль над состоянием объекта. Коротко: инкапсуляция скрывает внутренние данные и оставляет наружу понятный безопасный интерфейс.

## Вопрос 45. Контроль доступа к полям объекта (`public`, `private`)

Контроль доступа в C++ определяет, какие поля и методы класса доступны из разных частей программы. Модификатор `public` означает, что член класса доступен внешнему коду. Модификатор `private` означает, что доступ разрешён только внутри самого класса и его дружественных сущностей. Также существует `protected`, доступный внутри класса и его наследников.

Обычно поля объекта делают `private`, а методы, через которые объектом можно пользоваться, — `public`. Это помогает защитить объект от некорректных изменений. Например, внешний код не должен напрямую менять баланс счёта, если нужно проверять ограничения. Вместо этого он вызывает метод, который выполняет проверку и только затем изменяет состояние объекта. В C++ у `class` доступ по умолчанию `private`, а у `struct` — `public`, хотя оба механизма могут иметь методы, конструкторы и другие члены.

Разница между `public` и `private` важна для устойчивости программы. Если завтра внутреннее хранение баланса изменится, внешний код не должен переписываться, пока публичные методы остаются теми же. Типичная ошибка — думать, что `private` нужен только «для красоты»; на практике он защищает инварианты объекта. Коротко: `public` — интерфейс для внешнего кода, `private` — внутренняя реализация класса.

## Вопрос 46. Open‑Closed принцип

Open‑Closed Principle, или принцип открытости/закрытости, означает, что программные сущности должны быть открыты для расширения, но закрыты для изменения. Идея состоит в том, что новую функциональность лучше добавлять через расширение, наследование, композицию, полиморфизм или новые реализации интерфейса, а не через постоянное переписывание уже работающего кода.

Практический смысл принципа — уменьшить риск поломки старой логики. Если базовый класс или модуль уже используется в программе, его изменение может создать ошибки в существующем коде. Поэтому лучше проектировать систему так, чтобы новое поведение добавлялось отдельными классами или методами, не нарушая уже проверенную часть программы. Например, вместо большого `switch` по типу фигуры можно сделать общий интерфейс `Shape` с виртуальным методом `area()` и добавлять новые фигуры отдельными классами.

Важно понимать принцип не буквально: код иногда нужно менять, особенно при исправлении ошибок или изменении требований. Смысл OCP в том, чтобы стабильная часть системы меньше зависела от каждого нового варианта поведения.

## Вопрос 47. Конструкторы и деструкторы

Конструктор — это специальный метод класса, который вызывается при создании объекта и подготавливает его начальное состояние. Он имеет имя класса, не имеет возвращаемого типа, может принимать параметры, задавать значения полей, выделять ресурсы и проверять корректность начальных данных.

Деструктор — специальный метод, который вызывается при уничтожении объекта. Его имя начинается с `~`, он не имеет возвращаемого типа и параметров. Деструктор нужен для освобождения ресурсов: памяти, файлов, сетевых соединений, дескрипторов устройств. В C++ конструкторы и деструкторы особенно важны, потому что позволяют связать жизненный цикл объекта с управлением ресурсами. На этом основан принцип RAII.

Неочевидный момент: конструктор не «возвращает объект», а инициализирует уже создаваемый объект. В C++ поля лучше инициализировать в списке инициализации конструктора, особенно если это `const`-поля, ссылки или объекты без конструктора по умолчанию. Если объект локальный, его деструктор вызывается автоматически при выходе из области видимости. Для объектов, созданных динамически, освобождение должно происходить через правильный механизм владения, в современном C++ чаще через умные указатели.

При наследовании конструкторы вызываются от базового класса к производному, а деструкторы — в обратном порядке: сначала уничтожается производная часть объекта, затем базовая. Если класс предназначен для полиморфного использования через указатель или ссылку на базовый класс, у базового класса обычно должен быть виртуальный деструктор. Это нужно, чтобы при удалении объекта производного класса через указатель на базовый корректно вызвались все нужные деструкторы.

## Вопрос 48. Принцип RAII

RAII — это принцип управления ресурсами в C++, при котором ресурс должен жить столько же, сколько объект, который им владеет. RAII расшифровывается как Resource Acquisition Is Initialization — «получение ресурса есть инициализация». Смысл принципа в том, что ресурс захватывается при создании объекта и освобождается при его уничтожении.

Например, объект может открыть файл в конструкторе и закрыть его в деструкторе. Когда объект выходит из области видимости, деструктор вызывается автоматически, поэтому ресурс освобождается даже при исключениях или раннем выходе из функции. RAII помогает избежать утечек памяти и других ресурсов. В современном C++ этот принцип реализуют стандартные классы вроде `std::string`, `std::vector`, `std::unique_ptr`, `std::shared_ptr`, `std::lock_guard` и файловые потоки.

Пример идеи RAII без лишних деталей: `std::ofstream file("log.txt");` открывает файл при создании объекта, а при выходе из области видимости деструктор закроет файл автоматически. Типичная ошибка — вручную выделять ресурс и забывать освободить его на одном из путей выхода из функции. Коротко: RAII связывает ресурс с объектом, чтобы ресурс освобождался автоматически при уничтожении объекта.
